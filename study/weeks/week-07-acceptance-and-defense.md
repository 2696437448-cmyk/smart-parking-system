# 第 7 周：验收、报告与答辩表达

## 本周目标

- 理解这个项目是如何证明自己“做完了”的
- 看懂验收脚本、OpenAPI、报告、交付物之间的关系
- 学会把源码阅读结果转成答辩表达
- 能按最终页面顺序完成一套 10 到 15 分钟的展示流程

## 你会先看哪些文件

- `openapi/smart-parking.yaml`
- `scripts/defense_demo.sh`
- `scripts/test_step40_release_acceptance.py`
- `reports/step40_technical_acceptance.md`
- `reports/thesis_evidence_package.md`
- `docs/defense_demo_runbook.md`
- `deliverables/README.md`

## 这部分代码到底在做什么

很多课程项目只做到“代码能跑”。  
但这个项目还做了另外几层东西：

- `OpenAPI`
  说明接口合同是什么
- `acceptance scripts`
  说明系统如何自动验证
- `reports`
  说明每一步结果怎么记录
- `deliverables`
  说明最终怎么打包交付

这意味着它不仅是一个代码仓库，也是一个“可展示、可验收、可答辩”的项目。

### OpenAPI 在这里的意义

它不是摆设。  
特别是 owner/admin dashboard 接口，在 Step38 之后已经被正式纳入合同。  
这说明前后端不只是“口头约定”，而是有明确的接口结构。

### 验收脚本在这里的意义

验收脚本解决的是：  
项目不能只靠肉眼说“好像能跑”，还要有一套能重复执行的检查方式。

### 报告在这里的意义

报告不是重复 README，而是为了回答：

- 做了什么
- 怎么验证
- 结果如何
- 当前完成态到哪一步

## 一条必须跟读的调用链

这周跟一条“代码到验收”的链：

1. 开发完成 Step40 当前主线
2. `test_step40_release_acceptance.py` 负责执行默认验收
3. `defense_demo.sh` 提供演示和验收入口
4. `reports/step40_technical_acceptance.md` 记录结果
5. `deliverables/` 负责最终交付包

## 给新手的概念解释

### 什么是接口合同

接口合同就是“前后端都认的一份数据约定”。  
比如请求参数叫什么、返回结构里必须有哪些字段。  
这个项目里主要用 `openapi/smart-parking.yaml` 来表达。

### 什么是 acceptance

acceptance 可以理解成“最终验收”。  
它不是单个单元测试，而是验证系统整体功能和完成态的一组检查。

### 为什么答辩项目需要报告

因为答辩不是只展示页面，还要解释：

- 架构为什么这么设计
- 如何证明系统能工作
- 哪些改动是在哪一步完成的

## 一套可直接照着走的展示流程

下面这套流程默认你使用 `Step40` 当前完成态，适合 10 到 15 分钟展示。

### 演示前 1 分钟：准备动作

1. 先确认 Docker Desktop 已启动。
2. 在项目根目录执行：

```bash
./scripts/defense_demo.sh preflight
./scripts/defense_demo.sh start
```

3. 打开两个页面：
   - `http://localhost:4173/owner/dashboard`
   - `http://localhost:4173/admin/monitor`
4. 备用页只保留一个：
   - `http://localhost:8080/actuator/health`

### 第 1 段：先讲系统目标，不急着点页面

先用一句话把项目讲清楚：

“这个系统面向智慧社区停车场景，把多源数据处理、需求预测、车位调度、预约主链、导航和物业监管页面放进了一套完整系统里。业主端强调停车旅程闭环，物业端强调经营视图和状态感知。”

这样开场的好处是，老师后面看页面时会知道你展示的不是几个分散功能，而是一条完整业务链。

### 第 2 段：展示业主端首页

打开 `/owner/dashboard`，重点按下面顺序讲：

1. 先指左侧切换区和顶部页头。
   这里要强调：当前版本把 `首页 / 订单 / 导航` 三段式页面固定下来，用户先知道自己在哪一页。
2. 再指参数表单。
   说明用户可以按区域和时间窗发起推荐请求。
3. 最后指推荐卡片区。
   说明每张卡同时给出车位、区域、预计费用、预计到达时间和预约动作。

这一段你最好用一句总结收口：

“业主端首页不是只放推荐结果，而是把任务摘要、预约参数和推荐结果整合到同一个入口页里。”

### 第 3 段：展示订单页

从业主端首页进入订单页，重点讲三点：

1. 订单页承接同一个 `orderId` 上下文。
2. 页面展示订单编号、计费状态、金额和结算动作。
3. 用户不需要返回首页重新确认预约状态。

这里建议你用“连续性”这个词：

“订单页不是孤立页面，它是业主停车旅程里的第二段，负责把预约结果、费用信息和结算动作连续接住。”

### 第 4 段：展示目标车位导航

继续进入导航页，重点看：

1. 目标车位和区域标签
2. ETA 与路线摘要
3. 页面内地图预览
4. 外部地图跳转入口

这里的核心话术是：

“导航页继续复用同一个订单上下文，所以推荐、订单、导航不是三个分散页面，而是一条完整业务闭环。”

### 第 5 段：展示物业端监管总览

切到 `/admin/monitor`，按这个顺序讲：

1. 先讲顶部摘要板。
   运行模式、数据来源、实时占用率、活动预约数、今日收益、调度策略都在这里。
2. 再讲右侧说明卡片。
   这里强调最近更新时间、错误信息、覆盖区域数量、收益采样点和预测点位。
3. 最后讲图表区。
   说明页面不是图表堆叠，而是围绕收益和占用率这些管理问题组织的。

这段最重要的一句是：

“物业端不是普通监控页，而是一个一屏监管总览页，先帮助管理人员形成整体判断，再展开细节分析。”

### 第 6 段：讲实时通道和降级，而不是只讲好看的 UI

这时要把技术解释接上：

1. Admin 页面一方面消费 dashboard 聚合接口。
2. 另一方面尝试通过 WebSocket 获取实时状态。
3. 如果实时服务不可用，页面会自动切到 polling。
4. 页面会通过状态标签告诉你当前是实时、降级还是待刷新。

这段是老师最容易觉得“有工程含量”的地方。  
建议直接讲：

“这里不是单纯把页面做漂亮，而是把实时链路、降级链路和状态表达都收到了页面里。”

### 第 7 段：用接口和验收证明不是只会点页面

页面展示完后，不要马上结束。  
接着补两句技术证据：

1. owner/admin dashboard 已经进入 `OpenAPI` 合同。
2. 默认完成态可以通过 `Step40` 验收脚本重复验证。

如果现场允许跑命令，就执行：

```bash
./scripts/defense_demo.sh acceptance
```

如果时间不够，就至少口头说明：

“这个项目不是靠肉眼说能跑，而是有默认的综合验收入口来证明当前完成态。”

### 第 8 段：结束时的收口话术

最后建议用这段收口：

“我的系统最终形成了两条清楚的展示主线。业主端体现的是首页、订单、导航三段式连续停车旅程；物业端体现的是经营摘要、实时状态、趋势图表和降级提示组成的一屏监管总览。再往后用 OpenAPI、验收脚本和交付包把这套完成态固定下来，所以它不只是页面演示，而是一个可验证、可交付的毕业设计系统。”

## 本周可直接运行的命令

```bash
# 先做最小预检
./scripts/defense_demo.sh preflight

# 启动环境
./scripts/defense_demo.sh start

# 运行默认 Step40 验收
./scripts/defense_demo.sh acceptance

# 或直接运行默认验收脚本
python3 scripts/test_step40_release_acceptance.py

# 构建交付包
make release-bundle

# 展示结束后清理
./scripts/defense_demo.sh stop
```
