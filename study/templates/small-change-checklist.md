# 小改动检查清单

这份清单适合你在第 8 周或者以后准备做一个小改动时使用。  
目标不是让你一次改很多，而是确保你每次都改得小、改得准、改完能验证。

## 开始前先判断

先回答这 4 个问题：

1. 这是页面展示问题，还是数据结构问题？
2. 这是前端改动，还是后端 view-model 改动？
3. 这是当前主线文件，还是历史保留文件？
4. 改完后我准备用什么方式验证？

如果你连第 1 个问题都答不上来，先不要改，先回去追调用链。

## 新手优先选择的改动类型

优先改：

- 文案
- 状态提示
- 已有字段展示顺序
- 页面上已有数据的表达方式
- dashboard 已有字段的小调整

暂时别先改：

- 并发一致性逻辑
- 模型核心算法
- 网关熔断参数
- ETL 主流程结构

## 改动前

确认这些事情：

- 我知道入口文件在哪
- 我知道调用链至少有 3 步
- 我知道当前真实主线文件是什么
- 我知道这个改动最小会影响哪个页面或接口

## 改动中

保持这些原则：

- 一次只做一小块
- 不要同时改前端和后端太多地方
- 先改最接近问题的那层
- 如果只是显示问题，优先别去改深层业务逻辑

## 改动后最小验证

至少做下面 3 项里的 2 项：

1. 页面重新打开确认效果
2. 对应服务重新运行或构建
3. 运行最接近的验收或检查命令

例如：

```bash
cd apps/frontend
npm run typecheck
npm run build
cd ../..

python3 scripts/test_step40_release_acceptance.py
```

## 改动后复盘

改完后给自己写 4 句话：

1. 我改了什么
2. 为什么改
3. 影响了哪条链路
4. 我是怎么验证的

## 一个最小示例

如果你发现 Admin 页面里某个说明文案不清楚：

1. 先找 `apps/frontend/src/pages/AdminMonitor.vue`
2. 再看它依赖的 `useAdminDashboardView.ts`
3. 判断这是纯文案问题还是数据问题
4. 如果只是文案问题，只改页面或状态提示层
5. 改完后跑 `npm run build`

## 给自己的提醒

你的目标不是一上来证明自己能改大需求。  
你的目标是先建立一种稳定的方法：

- 我知道该从哪找
- 我知道该先改哪
- 我知道改完怎么验
