# 第 4 周：小改动、验证与答辩表达

前三周的重点是“看懂”。  
这一周要往前迈一步，变成“我能选一个小地方改一下，而且改完我知道怎么证明它没坏”。

## 这周只学什么

只回答 3 个问题：

1. 新手第一次改项目，优先改哪类问题
2. 改动前为什么一定要先追调用链
3. 改完后为什么必须做最小验证和讲解复盘

## 这周配套讲义

按这个顺序读：

1. [第 8 周：新手可做的小改动](../weeks/week-08-small-changes.md)
2. [小改动检查清单](../templates/small-change-checklist.md)
3. [每周笔记模板](../templates/weekly-note-template.md)

## 这周优先做什么改动

只从这些里选一个：

- 页面文案
- 页面状态提示
- 已有字段展示顺序
- dashboard 已有字段的展示表达
- 某个页面里已经存在的数据呈现优化

## 这周先不要碰

- 并发一致性主链
- 订单关键状态机
- 模型核心算法
- 网关熔断策略
- ETL 主流程结构

## 带读节奏

### 第 1 次：选一个最小改动题

先从页面入手，优先考虑：

- `OwnerDashboard.vue`
- `OwnerOrders.vue`
- `OwnerNavigation.vue`
- `AdminMonitor.vue`

判断问题时只问：

1. 这是显示问题，还是数据来源问题？
2. 这是前端能解决，还是要动后端 view-model？
3. 改完后我准备用什么验证？

### 第 2 次：追改动影响链

至少写出这条最短链：

1. 问题出现在什么页面或接口
2. 页面依赖哪个 composable / service
3. 如果涉及后端，是哪个接口或 view-model 提供数据

如果你连这条链都说不清，不要急着改。

### 第 3 次：做最小改动和最小验证

改动之后，至少做其中 2 项：

```bash
cd apps/frontend
npm run typecheck
npm run build
cd ../..

python3 scripts/test_step40_release_acceptance.py
```

如果改的是接口展示或 view-model，也可以补接口验证：

```bash
curl "http://localhost:8080/api/v1/owner/dashboard?location=R1&preferred_window=2026-03-31T09:00:00/2026-03-31T10:00:00&user_id=owner-app-001"
```

### 第 4 次：写答辩式复述

这一步非常重要。  
你不只是要“改过”，还要能讲出来：

1. 我改了什么
2. 为什么改这个
3. 我怎么定位到它
4. 影响了哪条链
5. 我怎么验证的

## 这周必须会说的主链

1. 先定位问题所在页面或接口
2. 追到最短调用链
3. 判断改前端还是改 view-model / 接口层
4. 做一次最小改动
5. 跑最小验证
6. 用自己的话复述改动价值和验证结果

## 这周复述题

1. 为什么新手第一次改动要尽量小？
2. 为什么展示问题通常不需要先动深层业务逻辑？
3. 改动后为什么不能只靠“我看起来觉得没问题”？

## 这周定位题

1. 找出一个你最想改的小点，并写出它对应的页面或接口入口。
2. 找出这个改动最短会影响哪条链。
3. 找出你要用哪两个命令做最小验证。

## 这周完成标准

- 你选出了一个适合新手的最小改动点
- 你写出了改动前的最短调用链
- 你完成了最小验证方案
- 你能讲出一段 2 分钟的“我改了什么、怎么验证”的说明

## 最终交付建议

这一周结束时，建议你至少留下一份自己的小结，内容包含：

- 改动目标
- 涉及文件
- 调用链
- 验证命令
- 结果截图或结果说明

这样你后面无论是继续开发，还是做课程答辩，都会更有底。
