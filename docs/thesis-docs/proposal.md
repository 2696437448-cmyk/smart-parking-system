## 二、研究意义、研究现状、研究内容、拟采用的研究思路与方法（可附页）

### （1）研究意义

随着城市机动车数量持续增加，社区停车难已经成为居民日常生活中比较常见的问题。很多时候并不是绝对没有车位，而是车位状态更新不及时、共享信息不透明、临时停车请求缺少统一调度，结果造成一部分住户反复找车位，另一部分车位又没有被及时利用。对业主来说，这会直接影响停车体验；对物业来说，也会增加管理成本和沟通成本。

从课题性质来看，智慧停车并不是单一的页面展示问题，也不是单纯的算法问题。它既涉及车位状态、车辆进出和住户出行规律等多源数据的整合，也涉及预约、计费、导航、经营分析等业务流程，还涉及系统在并发、异常和实时状态切换下能否稳定运行。如果只做静态查询，很难真正改善停车效率；如果只做预测或调度算法，而没有完整业务闭环，也难以形成一套可以演示和验收的毕业设计成果。

因此，本课题尝试把多源数据融合、轻量级预测、动态调度和工程实现结合起来，设计一套面向社区场景的智慧停车系统。该课题既贴近现实需求，也能够体现数据科学与软件工程的结合，对本科毕业设计来说具有较强的实践意义。

### （2）研究现状

目前，智慧停车相关研究主要集中在三个方面。第一类是停车信息采集与基础管理，重点解决车位状态感知、车辆出入记录和后台信息展示问题。第二类是停车需求预测，通常利用时间序列分析或机器学习方法估计区域车位占用变化，为推荐和管理决策提供参考。第三类是资源分配与调度优化，主要研究在多种约束下如何提高车位匹配效率和资源利用率。

从系统实现角度看，前后端分离和微服务架构已经成为比较常见的技术路线。后端多使用 Java 生态框架承载业务服务，数据库、缓存和消息机制用来支持订单、状态与统计信息管理；前端则普遍采用现代 Web 框架完成交互展示，并结合地图、图表等组件提升可视化效果。这说明相关研究已经不再停留在单纯的信息展示层面，而是逐步向业务闭环和平台协同方向发展。

不过，现有研究和系统实现仍然存在一些不足。第一，不少工作更强调某个单一技术点，例如只关注预测精度，或者只关注调度算法本身，缺少从数据接入到业务执行再到结果反馈的整体链路设计。第二，一些系统虽然功能较多，但在一致性控制、服务降级、实时状态切换和证据留存等工程问题上的考虑还不够充分。第三，部分课题可以完成若干功能模块，却没有形成清晰的论文结构和可复现实验支撑。基于这些情况，本课题选择从“多源数据融合 + LSTM-Lite 预测 + Hungarian 动态调度 + 业务闭环实现”的角度切入，重点建设一套适合社区场景、可部署、可展示、可验证的智慧停车系统。

### （3）研究内容

本课题拟面向社区智慧停车场景，设计并实现一套同时服务于业主端和物业端的车位动态调度与共享系统，重点包括以下内容。

第一，完成多源数据融合与预处理设计。系统将车位状态数据、车辆进出事件数据和住户出行规律数据作为主要输入来源，通过清洗、字段对齐和特征构建形成预测与调度所需的数据基础。

第二，完成停车需求预测模块设计。针对停车占用变化具有时间相关性的特点，系统采用轻量级 LSTM-Lite 方案，对供需缺口和区域占用趋势进行估计，为车位推荐和经营分析提供辅助信息。

第三，完成车位动态调度模块设计。系统基于 Hungarian 匹配思路，在考虑车位可用状态、区域信息和业务约束的前提下，对请求与车位资源进行动态分配，提高车位利用率并保证调度过程具有一定可解释性。

第四，完成业主端业务闭环设计。业主端主要包含推荐、预约、订单、计费和导航等页面，使用户从进入系统到完成停车能够形成较连贯的操作流程。

第五，完成物业端经营监控与可视化设计。物业端重点展示经营摘要、收益趋势、占用率趋势、预测对照以及实时状态说明，使管理人员既能看到业务结果，也能了解系统当前运行情况。

第六，完成系统可靠性与验收方案设计。考虑到停车业务对稳定性和一致性要求较高，系统在实现过程中将围绕请求幂等、锁控制、消息重试、故障降级、实时通道回退和综合验收等内容展开设计。

### （4）拟采用的研究思路与方法

本课题拟采用“文献分析、需求分析、系统设计、模块实现、实验验证”相结合的研究思路。

首先，在文献分析阶段，围绕智慧停车、停车需求预测、资源调度、多源数据融合、微服务架构和可视化管理等方向查阅资料，梳理相关系统在功能实现和工程落地方面的常见思路，为课题设计提供理论参考。

其次，在需求分析阶段，从社区停车场景中的实际问题出发，对业主端和物业端两类角色进行功能拆分，明确系统需要解决的核心问题，包括推荐是否及时、预约流程是否连贯、调度是否合理、计费是否清晰、管理端是否便于监控以及异常情况下系统能否保持基本可用。

再次，在系统设计阶段，采用分层与模块化相结合的方法完成总体架构设计。后端以网关、停车主业务服务、模型服务和实时服务为核心，前端采用页面化与组件化方式组织界面，数据库、缓存和消息机制共同支撑业务闭环与状态同步。

随后，在模块实现阶段，围绕数据预处理、需求预测、动态调度、预约计费、导航展示、经营看板和实时状态说明等模块逐步完成实现。实现过程中尽量选择适合本科毕业设计的技术方案，强调功能完整、逻辑清楚和结果可复现，而不是过度追求复杂堆叠。

最后，在实验验证阶段，从功能正确性、业务闭环完整性、系统稳定性和结果可解释性等方面对系统进行检验。验证内容包括数据处理结果、预测与调度服务运行情况、预约与计费流程表现、实时通道与降级行为以及综合验收结果等，以此验证课题方案的可行性。
