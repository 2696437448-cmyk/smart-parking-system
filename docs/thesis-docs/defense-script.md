## 正式答辩稿

各位老师好，我的毕业论文题目是《智慧社区多源数据融合的车位动态调度与共享系统设计》。下面我从课题背景、研究目标、系统架构、关键实现、实验结果以及不足与展望几个方面进行汇报。

### 一、课题背景与研究意义

我在做这个题目时，最先关注的是社区停车里的一个现实问题。很多时候，住户觉得停车难，并不一定是因为完全没有车位，而是因为车位信息更新不及时、共享车位调度不顺、停车流程不够连贯。结果就是车主在场内反复找位，物业也难以及时掌握真实的资源使用情况。

如果系统只能提供一个静态空位列表，作用其实比较有限。因为社区停车会受到时间段、出入记录、住户出行习惯和实时状态变化的影响，所以它更像是一个需要数据、调度和业务流程一起配合的问题。也正因为这样，我希望这个课题不只停留在页面展示上，而是尽量做成一套从数据处理到业务执行都能连起来的系统。

这项工作的意义主要有两点。第一，它有比较明确的现实背景，贴近日常生活中的停车问题。第二，它能够把多源数据融合、需求预测、调度优化和软件工程实现放在同一个课题里，对本科毕业设计来说比较完整，也比较适合展示自己的综合实践能力。

### 二、研究目标与主要内容

本课题的总体目标，是设计并实现一套面向社区场景的智慧停车调度与共享系统，使系统能够完成从数据接入、需求预测、资源调度到业务执行和状态展示的完整闭环。

围绕这个目标，我主要做了六方面的工作。

第一，整理多源数据输入，把车位状态、车辆进出记录和住户出行规律等信息结合起来，为后续预测和调度提供基础。

第二，设计停车需求预测模块。这里采用的是轻量级 LSTM-Lite 方案，重点是让预测结果能够稳定进入后续业务流程，而不是单纯追求模型复杂度。

第三，设计动态车位调度模块。系统基于 Hungarian 匹配思路，把用户请求和可用车位进行动态分配，提高资源利用效率。

第四，打通业主端业务流程，包括推荐、预约、订单、计费和导航，使用户能够完成较完整的停车操作。

第五，完成物业端经营监控页面，展示经营摘要、收益趋势、占用率变化、预测对照和实时状态说明。

第六，补充系统可靠性设计，比如幂等控制、锁机制、消息重试、服务降级和实时回退，使系统在异常场景下也能保持基本可用。

### 三、系统总体架构

本系统采用微服务加前后端分离的总体架构。后端主要由四个服务组成，分别是网关服务、停车主业务服务、模型服务和实时服务；前端则负责业主端和物业端页面展示。

其中，网关服务主要负责统一入口、请求转发、链路追踪和降级兜底。停车主业务服务负责预约主链、共享计费、订单处理、收益统计、导航目标生成以及 dashboard 聚合接口。模型服务负责预测、优化和模型版本管理。实时服务负责 WebSocket 推送以及实时状态相关能力。前端使用 Vue3 和 TypeScript 构建页面，并结合 Leaflet 地图和 ECharts 图表完成交互展示。

从业务流角度看，原始车位状态、车辆进出和住户出行数据先进入数据处理链路，形成预测和调度所需的基础数据。随后，业主请求通过网关进入停车主业务服务，在幂等控制、锁机制和数据库约束的共同作用下完成一致性处理。模型服务提供预测和调度支撑，实时服务负责状态更新，最后结果反馈到业主端和物业端页面中。

### 四、核心模块与关键技术

我觉得这个系统里有五个比较关键的点。

第一，多源数据融合。系统不是只看某一张车位表，而是把车位状态、车辆进出记录和住户行为规律结合起来，这样得到的数据基础会更完整。

第二，预测和调度是结合在一起用的。预测部分采用 LSTM-Lite，对供需变化做一个轻量估计；调度部分采用 Hungarian 方法进行动态匹配。这里的重点不是把算法讲得多复杂，而是让它们能真正服务推荐、经营分析和车位分配。

第三，系统做了完整业务闭环。业主端从推荐到预约，再到订单、计费和导航，可以连起来使用；物业端则可以看到经营摘要、趋势图和实时状态说明。

第四，系统考虑了可靠性设计。停车预约场景里如果并发控制做不好，很容易出现重复处理或者资源冲突。所以系统在主业务服务中加入了幂等控制、分布式锁和数据库约束，同时也给实时链路和部分服务调用准备了降级和回退方案。

第五，前端表达方式也做了整理。后期我把页面的数据编排继续往 view-model 层下沉，并统一处理 loading、error、empty、degraded、stale 等状态，这样页面在正常和异常两种情况下都更清楚，答辩展示时也更容易说明问题。

### 五、实验与验收结果

在实验和验证方面，我不是只做了单个页面演示，而是尽量结合项目阶段报告和综合验收结果来说明系统能力。

在数据与模型方面，系统完成了数据质量检查、特征处理、模型训练与对比、模型注册和激活等工作，为预测与调度链路提供了基础支撑。在一致性和可靠性方面，系统围绕预约主链、异常降级、消息重试、实时状态切换和监控展示进行了专项验证，说明系统在异常场景下仍然能保持基本可用。

在项目后期，系统又补齐了 dashboard 接口文档、页面级数据编排、聚合层拆分和图表按需加载等内容，并完成了综合技术验收。从最后结果来看，历史稳定能力保持有效，新增加的 owner/admin dashboard 收敛、前端状态统一和性能优化也通过了检查。整体上，系统已经具备较完整的业务闭环和答辩证据基础。

### 六、创新点与实际价值

我认为这个课题的特点主要体现在以下几个方面。

第一，它不是只做一个算法模块，也不是只做一个前端页面，而是把多源数据融合、预测、调度、预约计费导航和物业经营监控放在同一套系统里完成。

第二，系统同时面向业主端和物业端。业主端强调停车流程是否顺畅，物业端强调经营监控和状态解释，这样更贴近真实使用场景。

第三，课题在工程实现里比较重视稳定性问题。除了功能本身，还考虑了幂等控制、异常降级、实时回退和接口合同收敛，这使系统不只是“能展示”，而是更接近一个完整工程项目。

第四，系统已经形成了比较清楚的验收材料和文档证据，这对本科毕业设计来说是有实际价值的，因为结论不是空写出来的，而是可以回到项目材料中对应查看。

### 七、不足与改进方向

当然，这个课题也还有一些不足。第一，当前实验规模和复杂场景覆盖还比较有限，整体上更适合社区级演示和工程验证。第二，模型部分采用的是轻量实现路线，后续还可以继续补充更细的特征和更系统的评估。第三，调度策略目前更强调可解释性和确定性，如果后续增加更多业务约束，还可以继续扩展。第四，论文表达方面也还可以继续加强与相关研究的横向比较，让研究定位更加清楚。

如果后面还有继续完善的时间，我会重点考虑四件事：补充更细的实验矩阵，增强多场景数据验证，继续优化物业端图表说明，以及在不影响系统可复现性的前提下，进一步提升模型和调度模块的适应性。

## 常见追问准备

### 1. 为什么选择这个题目？

因为社区停车问题比较贴近现实生活，同时又能把数据处理、调度优化和软件工程实现结合起来，比较适合作为综合型毕业设计课题。

### 2. 为什么没有采用更复杂的深度学习模型？

本课题更强调系统整体可实现、可部署、可复现。轻量级方案更适合本科毕业设计的时间条件，也更容易和后续调度及业务流程结合。

### 3. Hungarian 算法在系统中的作用是什么？

它主要用于在用户请求和可用车位之间建立匹配关系，帮助系统在一定约束下完成动态分配。选择它的原因是结果相对稳定，也比较容易解释。

### 4. 这个系统的创新点主要体现在哪里？

创新点不在于某一个技术特别新，而在于把多源数据融合、预测、调度、业务闭环和工程可靠性整合到同一个社区智慧停车系统中，并完成了可演示、可验收的整体实现。

### 5. 你的实验结果主要说明了什么？

实验和验收结果主要说明系统不仅实现了基本功能，而且在一致性控制、故障降级、实时状态更新和综合交付方面具备较好的可验证性，能够支撑论文结论。

### 6. 目前最大的不足是什么？

最大的不足还是实验规模和复杂场景覆盖有限，目前更偏向社区级演示和工程实现验证，距离更大规模、更复杂约束下的真实场景还有一定差距。
