# Smart Parking Shadcn 风格产品化 UI 设计

- 日期：2026-04-21
- 项目：`smart-parking-thesis`
- 设计状态：已确认方向，待用户审阅书面 spec 后进入实现计划
- 视觉方向：方案 A，克制的科技运营台

## 1. 背景

当前 `smart-parking-thesis` 前端已经完成一轮科技感重构，业主端与物业端具备统一的深色主题、驾驶舱式表达和基础动效，功能链路也能够通过现有前端验收脚本验证。但从最终成品视角看，这一版页面仍然更接近“做得比较好看的业务系统”，还没有完全达到成熟产品界面的层级感、秩序感和信息密度控制。

用户本轮希望参考 `shadcn/ui` 官方文档中的 `dashboard`、`sidebar`、`chart` 等页面组织方式，继续优化当前界面。这里的重点不是把项目切换到 `shadcn/ui` 本体，因为该体系以 React 为核心，和当前 `Vue 3 + Arco Design Vue` 工程并不直接兼容；真正需要吸收的是它的产品结构、卡片语义、标题层级、侧栏组织和图表容器语言。

因此，本轮设计的目标是：在不改变现有页面结构、不推翻现有技术栈、不破坏现有功能闭环的前提下，把当前科技风页面继续收束成更像成熟智能运营平台的最终界面。

## 2. 设计依据

本轮参考的主要公开资料如下。

1. `shadcn/ui` Blocks：强调页面级拼装能力，而不只是零散组件。
2. `dashboard-01`：展示成熟后台页面中 `sidebar + page header + KPI + chart cards` 的组织方式。
3. `Sidebar`：强调导航、品牌、辅助区和页面控制的结构化表达。
4. `Charts`：强调图表外层容器、标题、副标题、说明和 tooltip 的产品化组织。

本轮不会照搬这些页面的配色或组件 API，而是借用其以下设计原则。

1. 结构优先于装饰。
2. 页面层级优先于单个组件炫技。
3. 卡片应承载语义，而不是只充当背景盒子。
4. 主内容区、次级信息区、状态提示区的权重必须清晰。
5. 图表和地图必须被包装成页面主内容，而不是孤立技术控件。

## 3. 目标

本轮设计目标如下。

1. 保留现有页面结构和业务路径，继续沿用 `OwnerDashboard`、`OwnerOrders`、`OwnerNavigation`、`AdminMonitor` 四个核心页面。
2. 在现有 `Vue 3 + Arco Design Vue + ECharts + Leaflet` 技术栈上，做一次以产品化层级和成熟后台气质为核心的高级化重构。
3. 让物业端更像真正的“停车运营控制台”，而不是单页大屏或说明板。
4. 让业主端三页更像一套统一产品，而不是三张风格接近但结构松散的功能页。
5. 把标题区、KPI 卡、状态提示、图表卡、地图卡和任务卡统一成一套稳定的视觉语言。
6. 保持现有构建、类型检查和前端验收脚本可继续通过。

## 4. 非目标

以下内容不属于本轮范围。

1. 不切换到 React，不引入 `shadcn/ui` 组件源码。
2. 不重做路由，不新增页面，不改变现有 `/owner/*` 与 `/admin/*` 信息架构。
3. 不改动 dashboard 聚合接口合同，不重写现有主要 composable 业务语义。
4. 不为了视觉效果加入重型 3D、复杂粒子系统或喧宾夺主的大屏动画。
5. 不把系统改成浅色后台模板，也不把它变成纯展示型大屏。

## 5. 方案结论

本轮最终采用的方向是“结构学 `shadcn`，气质保留智能停车系统”。

与已经落地的 Arco 科技风版本相比，这一轮不是再次换库，而是建立在当前实现之上的第二层收束：

1. 外层布局更像成熟后台。
2. 页面头部更像产品 header，而不是大块展示 banner。
3. 卡片层级更清楚，减少同权信息块互相竞争注意力。
4. 图表和地图容器更像内容主卡，而不是功能嵌入块。
5. 发光边框、深色渐变和科技感仍然保留，但从“整体强发光”改为“局部重点高亮”。

## 6. 视觉系统

### 6.1 总体气质

整体气质定义为“克制的科技运营台”。第一眼应感受到这是一个成熟的智能平台，第二眼能够迅速分辨主次信息，第三眼才会注意到细节中的科技感和动态反馈。

这意味着页面不能继续沿用“大面积炫光 + 信息平铺”的表达方式，而要让留白、对齐、标题层级和卡片秩序先建立起来。

### 6.2 色彩

色彩继续使用深色主题，但收敛为更成熟的产品化配色。

1. 主背景采用石墨黑和深海军蓝的低对比渐变。
2. 主强调色使用冷青蓝，承担激活态、关键操作和实时状态。
3. 辅助强调色使用少量琥珀色，仅用于收益、金额、告警和重要提醒。
4. 弱化大面积高亮描边，只在选中导航、核心 KPI、主按钮和关键图表交互中使用更明显的光感。

### 6.3 版式与留白

版式上要吸收 `shadcn` 页面中最重要的一点：密度不是越高越高级，成熟后台的关键是秩序。

1. 页面一级区块之间拉开层级。
2. 一级卡片与二级卡片在边框、阴影和背景透明度上有明确区分。
3. 卡片内部使用更一致的标题、主值、说明、操作顺序。
4. 页面尽量形成“一个主焦点 + 两到三个辅助区”，而不是每个区块都争夺主视觉。

### 6.4 动效

动效仍保留 `@vueuse/motion`，但角色变更为“辅助聚焦”而不是“制造科技感”。

1. 页面进入时采用轻量向上浮现。
2. KPI 卡和图表卡采用小幅错峰出现。
3. hover、focus 和 active 态增加更细腻的亮度和边框反馈。
4. 禁止连续炫光扫动、无意义脉冲和大面积漂浮动画。

## 7. 布局策略

### 7.1 物业端

物业端是本轮改造的重点，整体布局采用“窄侧栏 + 顶部页面头 + 主内容区”的产品后台骨架。

左侧不再只是 `control rail` 说明栏，而是升级为更接近 `shadcn sidebar` 的结构：

1. 品牌与页面身份区。
2. 主导航区。
3. 运行状态与模式说明区。
4. 次级辅助区或说明区。

顶部不再使用大横幅式 banner，而是产品化 page header，主要承载：

1. 页面标题。
2. 当前模式与数据来源。
3. 最近更新时间。
4. 刷新和重连动作。

### 7.2 业主端

业主端不做完整后台侧栏，但要采用与物业端一致的“任务头 + 主内容区 + 次级导航”语言。

1. 顶部从展示型横幅升级为任务型 header。
2. 底部导航弱化按钮感，增强产品导航感。
3. 三个页面都共享一致的标题组件、状态标签、卡片语义和操作区组织方式。

## 8. 页面设计

### 8.1 AdminLayout

`AdminLayout.vue` 调整为更像成熟运营后台的壳层。

1. 左栏负责品牌、导航和运行状态。
2. 主区顶部为页面 header，而不是视觉型展示 banner。
3. 所有管理端页面后续都应复用这套外壳逻辑。

### 8.2 AdminMonitor

`AdminMonitor.vue` 继续承担“停车运营驾驶舱”职责，但重排主次层级。

推荐结构如下。

1. 第一层：核心 KPI 区。
2. 第二层：一个主图表卡和两个辅助经营卡。
3. 第三层：预测、区域收益、诊断摘要和降级说明。

这意味着原本同权并列的多个图表不再全部放在一个层级里，而是突出“当前最值得看的一张图”，其余图和解释区下沉为辅助信息。

### 8.3 OwnerLayout

`OwnerLayout.vue` 保留当前的顶栏与底部导航逻辑，但顶部重做为任务型 header。

1. 页面身份区体现“智能停车旅程”而不是单纯展示标题。
2. 当前状态、推荐能力、计费和导航能力以更精炼的标签形式出现。
3. 底部导航采用更轻的移动端产品导航表达，减少强按钮感。

### 8.4 OwnerDashboard

`OwnerDashboard.vue` 继续保留“摘要区 + 输入区 + 推荐区”，但编排要更像产品首页。

推荐布局为：

1. 顶部任务摘要头卡。
2. 左侧预约参数面板。
3. 右侧推荐结果卡组。

顶部摘要头卡不再只展示文案，而是组织成摘要带，收纳目标区域、候选车位、最近订单和计费单位，使首页一打开就能明确当前任务状态。

### 8.5 OwnerOrders

`OwnerOrders.vue` 调整为“订单摘要头卡 + 账单主卡 + 动作区”。

重点是让用户形成明确阅读顺序：

1. 先看当前状态。
2. 再看金额与时间节点。
3. 最后执行下一步动作。

状态 badge 不应只是附着在局部，而应形成更明确的页内状态带。

### 8.6 OwnerNavigation

`OwnerNavigation.vue` 保持“摘要信息卡 + 地图卡”双栏结构，但地图卡需要真正成为页面主视觉之一。

1. 摘要卡聚焦 ETA、目标车位、区域、路线摘要。
2. 地图卡拥有更清楚的标题、副标题和目标信息。
3. 外部地图入口作为清晰但次级的动作保留。

## 9. 组件设计

### 9.1 SectionHeader

`SectionHeader.vue` 将成为全站统一标题语言的核心组件。

统一结构固定为：

1. `eyebrow`
2. `title`
3. `subtitle`
4. `badge`
5. `action slot`

所有页面头卡、区块头和图表头都尽量在这套结构下统一。

### 9.2 MetricCard

`MetricCard.vue` 从单纯指标块升级为产品化 KPI 卡。

统一信息顺序为：

1. 标签或类别。
2. 主值。
3. 趋势、状态或辅助 badge。
4. 说明文字。

视觉上降低厚重阴影，改用更清晰的边框、背景层次和局部高亮。

### 9.3 ViewStateNotice

`ViewStateNotice.vue` 继续承担 `loading / ready / empty / error / degraded / stale` 语义，但提示方式要分层。

1. 默认使用轻量 inline 状态条。
2. 只有错误或强提醒场景才使用整块 notice。

目标是让状态存在但不过度打断页面阅读节奏。

### 9.4 EChartPanel

`EChartPanel.vue` 是物业端质感升级的关键。

图表卡应统一包含：

1. 标题。
2. 副标题。
3. badge 或时间范围。
4. 图表主体。
5. 底部说明。

同时，图表自身的线条、网格、tooltip 和 legend 也要更接近成熟产品的克制风格。

### 9.5 MapPreview

`MapPreview.vue` 要从功能嵌入块升级为内容主卡。

1. 地图上方保留目标和 ETA 摘要。
2. 地图本体获得更稳定的容器和标题层级。
3. 让用户感受到“这是页面主内容”，而不是一个附带能力。

## 10. 样式层改造范围

本轮样式改造的重点不只是换颜色，而是重建一套更接近 `shadcn` 页面组织逻辑的 token 和层级体系。

重点文件如下。

1. `apps/frontend/src/styles/tokens.css`
2. `apps/frontend/src/styles/base.css`
3. `apps/frontend/src/styles/layout.css`
4. `apps/frontend/src/styles/components.css`
5. `apps/frontend/src/styles/pages.css`

重点改造内容如下。

1. 背景层和面板层关系。
2. 一级卡片与二级卡片的边框、透明度和阴影体系。
3. 文字字号梯度和标题层级。
4. 导航、选中态、hover、focus 的统一规则。
5. 统计卡、任务卡、图表卡和地图卡之间的一致性。

## 11. 工程约束

为了避免视觉优化破坏现有完成态，本轮必须遵守以下约束。

1. 不改变既有页面的主业务顺序。
2. 不移除现有前端验收脚本依赖的结构性能力。
3. 不把页面实现改成高度耦合、只能为单页服务的样式堆叠。
4. 保持构建分包策略有效，不重新引入大 chunk 警告。
5. 保持桌面端和移动端都能正常使用。

## 12. 验收标准

本轮设计完成后的验收标准如下。

1. 物业端一眼看上去像“成熟运营后台”，而不是“科技风大屏”或“拼块式报表”。
2. 业主端三页具有统一产品语言，能明显看出是同一套系统。
3. 页面主次层级更清楚，用户能快速分辨当前页面最重要的区域。
4. 图表、地图和状态提示能自然融入页面，而不是突兀挂载的功能块。
5. `npm run typecheck`、`npm run build`、前端 UI 相关脚本继续通过。

## 14. 结论

本轮不是“再换一个 UI 库”，也不是“继续叠加科技特效”，而是在已经完成的 Arco 科技风版本上，吸收 `shadcn/ui` 成熟后台的组织方式，把当前项目真正收束成一套产品化的智能停车系统界面。

一句话概括本轮设计原则：

结构学习 `shadcn`，技术保持 `Vue + Arco`，气质保留智慧停车系统的深色智能感。
