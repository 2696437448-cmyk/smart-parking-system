# Smart Parking 双端科技感 UI 重构设计

- 日期：2026-04-20
- 项目：`smart-parking-thesis`
- 当前默认完成态：`Step40`
- 设计状态：已确认，待用户审阅书面 spec 后进入实现计划

## 1. 背景

`smart-parking-thesis` 前端当前已经具备完整业务链路，路由、页面、聚合接口、图表和地图能力都可以工作，但页面观感仍偏向“功能可用的业务系统”，距离“智慧停车中控系统”还有明显差距。用户最新要求不是继续做轻量样式优化，而是保留现有页面结构，在不改动后端合同和主业务语义的前提下，把业主端与物业端一起升级成更有科技感、智能感和成品感的最终版本。

这意味着本轮工作重点不再是补功能，而是建立一套完整的前端视觉系统：统一组件语言、统一状态表达、统一壳层风格，并让页面在第一次打开时就呈现出“智能停车系统界面”的感觉。与此同时，这套升级必须能安全落到现有 Vue 3 工程里，继续兼容 `ECharts`、`Leaflet`、`Pinia`、`Vue Router`、现有 composable 和 `Step40` 验收脚本。

## 2. 问题陈述

当前前端的主要问题集中在以下几个方面。

第一，双端页面虽然已经完成第一轮现代化整理，但视觉语义仍然偏弱。`OwnerLayout`、`AdminLayout`、各业务页和卡片组件之间还没有形成足够鲜明的“系统界面”层级，打开后更像普通后台页面，而不是智慧停车平台。

第二，现有样式体系虽然具备 `tokens/base/layout/components/pages` 分层，但主题仍偏明亮和克制，不足以支撑用户明确提出的“科技感、智能感”。如果继续只在现有主题上做局部修补，最终只会得到一个更整洁的业务系统，而不是一次真正有效的视觉升级。

第三，双端页面的组件外观和信息表达还没有完全统一。虽然 `MetricCard`、`SectionHeader`、`ViewStateNotice` 等组件已经存在，但它们还没有建立成一套足够强的视觉语言，缺少统一的按钮体系、标签体系、表单体系和玻璃面板体系。

第四，地图和图表已经是项目亮点，但它们当前更像“被放进页面里的功能组件”，不是驾驶舱里的核心显示单元。也就是说，展示能力有了，但外层界面还没有把它们包装成真正的系统面板。

## 3. 本轮目标

本轮重构的目标如下。

1. 在保留现有页面结构和路由的前提下，把业主端和物业端统一升级为深色蓝青科技风界面。
2. 引入稳定的 Vue 3 UI 组件库作为视觉底座，并与现有项目组件共存，不推翻已有业务语义组件。
3. 建立“壳层、导航、卡片、表单、标签、状态提示、图表容器、地图容器”统一风格。
4. 让业主端更像“智能停车助手”，让物业端更像“停车运营驾驶舱”。
5. 保持现有接口合同、页面路径、主要交互和运行逻辑不变，避免因视觉改造破坏 `Step40`。
6. 为后续论文与最终截图提供一套真正能代表最终系统版本的前端成品界面。

## 4. 非目标

以下内容不属于本轮范围。

1. 不改动后端聚合接口字段，不改写 owner/admin dashboard 合同。
2. 不新增新业务页面，不新增答辩专用大屏，不改变 `/owner/*` 与 `/admin/*` 路由结构。
3. 不更换 `ECharts` 与 `Leaflet` 的核心实现，不重写现有业务 composable 的主语义。
4. 不引入过重的 3D 引擎、粒子背景或喧宾夺主的炫技动画。
5. 不在本轮同时修改论文正文，论文更新要在最终 UI 冻结后再进行。

## 5. 方案比较

本轮设计阶段考虑了三条路线。

### 方案 A：Arco Design Vue + 深色科技主题 + 轻量动效层

这是最终选定方案。它的优点是与当前 Vue 3 工程兼容性好，组件覆盖面完整，适合后台、监控、表单和驾驶舱类页面。通过 `ConfigProvider`、组件替换和 token 定制，可以较快把项目从“自定义样式页面”升级为“成体系的智能系统界面”。同时，Arco 的组件气质比很多基础 UI 库更适合数据中台和运营类页面。

### 方案 B：PrimeVue + 自定义主题

PrimeVue 的上限较高，定制能力也很强，但在本项目当前结构里接入成本和视觉调校成本都更高。它适合从零建立完整设计系统，但本轮目标是保留现有结构并快速落地，PrimeVue 在此阶段性价比不如 Arco。

### 方案 C：继续沿用纯自定义组件，只强化 CSS

风险最低，但效果最受限。它无法从根本上建立按钮、输入、弹层、标签、骨架屏等组件语言的一致性，也难以让页面短时间内获得明显的科技感升级。因此该方案不满足用户预期。

最终选定方案 A。

## 6. 总体设计

本轮的核心原则是“页面结构不动，视觉系统重建，组件语义统一，双端一起升级”。

前端技术路线采用以下组合：

1. `Arco Design Vue` 作为基础组件库与主题底座。
2. `@vueuse/motion` 作为轻量动效层。
3. 保留项目现有业务语义组件，由它们继续承接停车领域表达。
4. 保留 `ECharts` 和 `Leaflet`，但重新包装其外层面板和主题语言。

页面组织保持当前结构：

1. `OwnerLayout` 仍然承接业主端入口。
2. `OwnerDashboard`、`OwnerOrders`、`OwnerNavigation` 三页继续保留。
3. `AdminLayout` 仍然承接物业端入口。
4. `AdminMonitor` 继续保留为核心运营监控页。

视觉系统采用“深海军蓝 + 冷青色高亮 + 少量橙色告警/金额强调”的策略。主题基座不再沿用当前偏暖色、偏浅底方案，而是整体切换到更强的深色中控风格，并通过半透明面板、发光描边、层次化数据卡和状态标签建立统一的系统界面感。

## 7. 视觉语言设计

### 7.1 色彩

主背景采用深海军蓝到蓝黑色的渐变体系，辅以低强度网格、雷达波纹或数据流线做氛围层。主强调色采用蓝青色，用于关键按钮、激活导航、数据高亮、实时状态和图表主曲线。橙色仅用于金额、关键业务动作和告警提示，不再作为大面积主色。

### 7.2 面板

所有主面板统一采用深色玻璃卡片逻辑：轻透明底、冷光边框、柔和发光投影、明确内边距和卡片层级。主面板与次级卡片必须能一眼区分，避免当前卡片在深色环境里全部压成同一层。

### 7.3 文本层级

标题、KPI 数值、说明文字、辅助标签要形成明显分层。KPI 数字和核心状态必须更亮、更稳、更具显示屏感觉；说明文字和辅助提示则保持低饱和与高可读性，避免深色主题下信息黏成一团。

### 7.4 动效

动效只允许服务于系统启动感和信息聚焦感。页面入场采用轻微上浮与渐显；卡片采用错峰入场；图表和地图采用延迟加载过渡。禁止大面积无意义漂浮、旋转或高频粒子效果，避免影响性能与专业感。

## 8. 双端页面设计

### 8.1 业主端总体定位

业主端应被重构为“智能停车助手”而不是普通移动业务页。用户进入业主端后，需要第一眼看到当前停车任务、推荐结果、计费摘要和下一步动作，而不是一堆平级信息块。

### 8.2 OwnerLayout

`OwnerLayout.vue` 保留顶部与底部导航结构，但整体样式改为中控导航条。顶部需要包含系统识别区、当前旅程说明和状态芯片，底部导航需要明显激活态与发光边界，使其更像车载或智能终端界面。

### 8.3 OwnerDashboard

`OwnerDashboard.vue` 保留“摘要区 + 输入区 + 推荐区”结构，但每一块都要升级为系统面板。

1. 摘要区改造成主任务面板，显示目标区域、计费单位、候选数量、最近订单与主操作。
2. 输入区基于 Arco 表单重构，形成统一深色科技表单区。
3. 推荐区改造成智能推荐卡阵列，每张卡除基础信息外，还要通过标签和版式强化“推荐依据、预计到达、当前区域、下一步动作”。

### 8.4 OwnerOrders

`OwnerOrders.vue` 改造成订单任务流面板。重点不是堆字段，而是让订单状态、金额信息、时间节点和操作区形成更明确的阅读顺序。金额区域要做重点突出，状态标签要具备清晰视觉等级，关键动作要形成单独操作条。

### 8.5 OwnerNavigation

`OwnerNavigation.vue` 保持“信息卡 + 地图区”结构，但整体要更像导航任务面板。左侧信息卡需要突出目标锁定、预计到达、距离和路线摘要；地图区要强化目标点和路线视觉焦点，让它成为整页真正的主视图之一，而不是一个附属地图控件。

### 8.6 物业端总体定位

物业端应被重构为“停车运营驾驶舱”。管理者进入页面后，要首先感受到这是一个实时业务系统，而不是一张静态报表页。

### 8.7 AdminLayout

`AdminLayout.vue` 保留左轨与主区域结构，但左轨要升级为系统控制轨道，包含身份区、能力标签、当前模式和主入口。头部 Banner 则要承担“运营中控台”的视觉中心作用。

### 8.8 AdminMonitor

`AdminMonitor.vue` 保留“摘要区 + 解释区 + 图表区”三层结构，但重排为更强的中控网格。

1. 顶部摘要区为高亮 KPI 区，实时占用率、今日收益、活动预约和调度策略必须具备强烈的数据屏质感。
2. 中部解释区作为运行解释层，负责展示更新时间、通道来源、降级状态、诊断入口和系统说明。
3. 下部图表区继续保留 4 张核心图表，但每张图都要有统一的中控面板头、深色图表主题和稳定的阅读节奏。

## 9. 组件层设计

本轮不推翻现有业务语义组件，而是采用“Arco 提供基础能力，项目组件继续承接业务含义”的模式。

### 9.1 继续保留并升级的项目组件

1. `MetricCard.vue`
2. `SectionHeader.vue`
3. `ViewStateNotice.vue`
4. `EChartPanel.vue`
5. `MapPreview.vue`

这些组件继续保留业务语义，但内部结构、标签、按钮、头部区域和视觉主题都将重新设计，以贴合 Arco 体系和深色科技主题。

### 9.2 新增基础库承担的职责

1. `a-button`、`a-space` 承担按钮与动作条基础能力。
2. `a-tag` 承担统一状态标签能力。
3. `a-form`、`a-input`、`a-select`、`a-date-picker` 或等价组件承担表单能力。
4. `a-card`、`a-statistic`、`a-alert`、`a-result` 作为卡片、统计与状态提示底座。

### 9.3 不直接替换的核心展示组件

`EChartPanel` 与 `MapPreview` 不直接被 UI 库替换，只升级外层皮肤和容器结构。原因是它们本身就是本项目的重要展示能力，应该继续保留业务控制力。

## 11. 实施顺序

实现必须按如下顺序推进。

1. 接入 `Arco Design Vue` 和深色科技主题底座。
2. 改造 `OwnerLayout` 与 `AdminLayout`，先让双端壳层统一成系统界面。
3. 升级 `MetricCard`、`SectionHeader`、`ViewStateNotice` 等共享组件。
4. 依次升级 `OwnerDashboard`、`OwnerOrders`、`OwnerNavigation`、`AdminMonitor`。
5. 最后统一 `EChartPanel` 与 `MapPreview` 的面板外观、主题和加载动效。
6. 全量执行 `typecheck`、`build` 和前端相关验收脚本。

## 12. 验收标准

本轮实现完成后，至少必须满足以下条件。

1. 双端页面整体观感明显变为深色科技中控风，不再保留当前浅色业务页主基调。
2. 业主端三页能明显呈现“智能推荐中心 / 订单任务流 / 导航任务面板”的系统感。
3. 物业端监控页能明显呈现“运营驾驶舱”的第一视觉印象。
4. 现有路由、页面结构、主要业务信息和主交互不被破坏。
5. `npm run typecheck` 通过。
6. `npm run build` 通过。
7. `ECharts` 与 `Leaflet` 页面仍然可用。
8. 移动端至少保持完整可用，不因深色重构造成基本交互损坏。

## 13. 风险与控制

第一类风险是“只换组件皮肤，没有形成系统感”。为避免这一点，本轮必须先改壳层，再改页面，而不是只替换按钮和输入框。

第二类风险是深色主题下对比度失控。为避免这一点，主题 token 需要先稳定字体颜色、边框亮度、表单焦点色和卡片层级，再进入页面细节调整。

第三类风险是图表和地图与新主题脱节。为避免这一点，`EChartPanel` 和 `MapPreview` 必须在末尾统一纳入面板体系，而不是单独保留旧皮肤。

第四类风险是 UI 库引入后破坏当前工程节奏。为控制这一点，本轮只引入 `Arco Design Vue` 和轻量动效层，不同时扩展到新的状态管理或渲染框架。

## 14. 通过条件

当且仅当以下条件同时满足时，本设计视为完成并可进入实现计划阶段。

1. 用户确认本书面 spec。
2. spec 已写入仓库并完成自检。
3. spec 已单独提交，不与业务实现混杂。
4. 后续工作先进入实现计划，再进入具体代码改造。
