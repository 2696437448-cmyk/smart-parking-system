# 风险登记与应对

## R1 - AI 规划过度乐观

- 表现：算法核心或跨服务集成被低估工期。
- 应对：
  - 对算法/跨服务步骤预留 1.5 倍缓冲；
  - 编码前先列阻塞点；
  - 步骤拆分到“单目标可验收”。
- 触发重排条件：单步骤超过 1 天仍无法过闸门。

## R2 - memory-bank 维护负担

- 表现：文档过多导致上下文噪声，进度不更新。
- 应对：
  - 执行时优先读 `architecture.md` + 当前步骤；
  - 进度采用追加式记录，避免反复重写。
- 触发清理条件：提示长度明显影响执行质量。

## R3 - 数据源脏数据/不可用

- 表现：缺字段、时间戳异常、空值和重复率过高。
- 应对：
  - 强制 Step 0 数据健康检查；
  - 准备结构对齐的 fallback 样例数据。
- 切换 fallback 条件：首轮数据质量未达最低阈值。

## R4 - Git 流程漂移与凭证泄露

- 表现：
  - 步骤执行时跳过分支/PR/标签，导致不可回溯；
  - 误提交 `.env`、token、密码等敏感信息。
- 应对：
  - 执行 `git-workflow.md` 的强制闸门（branch/commit/PR/tag）；
  - 开启分支保护，主分支仅允许 PR 合并；
  - `.env` 与凭证类文件禁止入库，首推前进行敏感词扫描。
- 触发重排条件：
  - 任一步缺少 Git 证据；
  - 发现敏感文件或密钥泄露迹象。

## R5 - 主线能力进入代码但文档口径滞后

- 表现：
  - 登录入口、最终 UI、学习资料已经进入 `main`，但 memory-bank / README / runbook 仍沿用旧口径；
  - 代码主线和论文/答辩说明出现脱节。
- 应对：
  - 每次合入跨前端/文档的大分支后，优先补 `memory-bank`；
  - 把“默认入口”“页面命名”“验收脚本”和“学习资料入口”视为必须同步的文档字段。
- 触发重排条件：
  - 代码已经要求从 `/login` 进入，但文档仍写“直接访问 dashboard”；
  - `study/`、论文文档或答辩手册已经入库，但 memory-bank 仍没承认它们是主线组成部分。

## 初始质量阈值

1. 必填字段完整率 >= 98%
2. 重复率 <= 1%
3. 关键特征空值率 <= 5%
4. 时间戳解析成功率 >= 99%
